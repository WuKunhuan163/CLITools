# iCloudPD 优化与发现报告

## 1. 核心发现

### 方向设置的“陷阱” (Direction Ambiguity)
在当前环境的 `pyicloud` 版本中，`DirectionEnum` 的表现与直觉相反：
- **`ASCENDING` (升序)**: 实际上返回的是 **最新 (Newest)** 的照片（按创建时间降序）。
- **`DESCENDING` (降序)**: 返回的是 **最旧 (Oldest)** 的照片（按创建时间升序）。
- **优化结论**: 扫描最新照片或从最新开始备份时，必须显式设置方向为 `ASCENDING`。

### 资源收集的性能瓶颈 (Gathering Bottleneck)
- **旧方案**: 之前通过迭代全库（Iteration）来匹配 ID。对于拥有 10 万张照片的库，如果目标照片位于扫描方向的末尾（Worst Case），需要遍历几万个元数据，耗时可能长达 **40-60 分钟**。
- **突破性发现**: 我们发现了 iCloud CloudKit 的 **`lookup` 接口**。通过构造 `POST .../records/lookup` 请求，可以直接根据 `recordName` (即资源 ID) 获取完整的 `PhotoAsset` 对象（包含最新的下载 URL）。
- **优化效果**: 对于 200 张照片的收集，从之前的几分钟（甚至几十分钟）缩短到了 **1-2 秒**。

## 2. 缓存 (JSON Cache) 的价值
`photos_cache.json` 不仅仅是元数据的镜像，它是整个工具的“搜索引擎”：
1. **即时过滤**: 支持 `--regex`, `--since`, `--before` 等选项在本地毫秒级筛选出目标 ID，无需请求 API。
2. **断点续传**: 下载前通过本地文件系统对比，结合缓存 ID 识别已下载内容。
3. **极速定位**: 结合上述 `lookup` 接口，缓存中的 ID 成为了直接获取下载地址的“钥匙”，彻底摆脱了对全量扫描的依赖。

## 3. 增量保存机制
为了应对长时间扫描的不稳定性，实现了双重持久化：
- **按日切换触发**: 判定某日扫描结束即写入。
- **30秒定时器**: 定期强制同步内存数据到磁盘。

## 4. 后续建议
- **批量 Lookup**: 建议收集阶段采用每 100 个 ID 一批进行批量请求。
- **多库支持**: 在 Lookup 时需要带上正确的 `zoneID`，这已在缓存中记录。
